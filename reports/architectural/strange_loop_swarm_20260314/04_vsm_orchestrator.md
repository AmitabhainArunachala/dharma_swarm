## Angle
Treat the viable system as a single control plane centered on `SwarmManager`, with `SystemMonitor` as S3* audit, `SleepCycle` as quiet-hours S4/S5 consolidation, and `orchestrate_live.py` as a thin supervisor only. The repo already has most of S1-S5, but daemon viability is broken by split control paths and weak anti-oscillation in the live path.

## Concrete Changes
- Make [swarm.py](/Users/dhyana/dharma_swarm/dharma_swarm/swarm.py) the single daemon control plane by extracting the body of `SwarmManager.run()` into a canonical `async tick()` method. Both `SwarmManager.run()` and `run_swarm_loop()` should call that, not `_orchestrator.tick()` directly.
- Keep wake-mode sequencing inside one loop: `rescue_recent_failures() -> _orchestrator.tick() -> coordination_status()/spawn_coordination_tasks() -> _director_pulse() -> _tick_living_layers() -> record_success`. Those methods already exist in [swarm.py](/Users/dhyana/dharma_swarm/dharma_swarm/swarm.py).
- Add quiet-hours sequencing inside `SwarmManager`: `drain urgent/recovery only -> run SleepCycle.run_full_cycle() once per quiet window -> hold non-urgent dispatch until wake`. Reuse [sleep_cycle.py#L109](/Users/dhyana/dharma_swarm/dharma_swarm/sleep_cycle.py#L109), do not create a second sleep orchestrator.
- Keep `SystemMonitor` pure, but add a bounded consumer in `SwarmManager`, e.g. `_apply_health_control(anomalies)`, limited to reversible controls: suppress director pulses, widen retry backoff, rotate thread, or enter sleep after repeated anomalies.
- Persist anti-oscillation state under `state_dir`: last successful tick, last dream density, last shakti time, last sleep window, and anomaly persistence counters. Do this in existing swarm/orchestrator state, not a new meta-daemon.
- Unify daemon identity around one PID file and one startup path. Either `orchestrate_live.py` becomes the canonical daemon and owns `daemon.pid`, or it should stop killing/monitoring `daemon.pid`.

## Risks
- Unifying `orchestrate_live.py` onto `SwarmManager` will change current live-daemon behavior; first pass should be control-path unification, not feature expansion.
- A hard sleep gate can starve urgent recovery unless rescue and explicit high-priority work stay allowed during quiet hours.
- Wiring health into control can create new oscillations if single anomalies trigger action immediately; require persistence and cooldown.
- PID unification may affect CLI/TUI/operator assumptions if both `daemon.pid` and `orchestrator.pid` are in use.

## Priority
1. Canonicalize the daemon around one `SwarmManager` tick path and one PID identity.
2. Restore anti-oscillation guarantees in live mode: breaker success reset, persisted dream/shakti hysteresis, anomaly persistence.
3. Sequence quiet-hours sleep explicitly through `SleepCycle` with urgent-only exceptions.
4. Let `SystemMonitor` drive only bounded, reversible S3 controls after repeated anomalies.
5. Defer deeper system-R_V and universal-loop work until the daemon path is structurally coherent.
