## Blind Spots
- The domain is implicit and hard-coded into schema. `Proposal` assumes `component`, `change_type`, `diff`, and code-execution metadata; `ArchiveEntry` assumes code-change lineage; `FitnessScore` assumes one fixed eight-axis objective. That prevents “one engine, many evolving domains.” [evolution.py](/Users/dhyana/dharma_swarm/dharma_swarm/evolution.py#L89) [archive.py](/Users/dhyana/dharma_swarm/dharma_swarm/archive.py#L68) [archive.py](/Users/dhyana/dharma_swarm/dharma_swarm/archive.py#L95)
- The stage graph is duplicated instead of configured. `run_cycle` and `run_cycle_with_sandbox` differ mainly by executor choice, but the engine expresses that by copying loop logic. [evolution.py](/Users/dhyana/dharma_swarm/dharma_swarm/evolution.py#L1492) [evolution.py](/Users/dhyana/dharma_swarm/dharma_swarm/evolution.py#L1749)
- Score geometry is globally frozen. `normalize_fitness_weights`, `FITNESS_DIMENSIONS`, and `MetaEvolutionEngine._trajectory_loss_gradient()` all assume the same fixed dimensions, so a new domain cannot define its own score manifold without patching core code. [archive.py](/Users/dhyana/dharma_swarm/dharma_swarm/archive.py#L21) [archive.py](/Users/dhyana/dharma_swarm/dharma_swarm/archive.py#L35) [meta_evolution.py](/Users/dhyana/dharma_swarm/dharma_swarm/meta_evolution.py#L297)
- Diversity is code-specific. `MAPElitesGrid.compute_feature_coords()` bakes in `dharmic_alignment`, `elegance`, and diff-size `complexity`, which is not a universal feature space. [archive.py](/Users/dhyana/dharma_swarm/dharma_swarm/archive.py#L167)
- Selection is artifact-type blind. `select_parent()` only selects applied archive entries and has no path for selecting domain genomes, recognition artifacts, or catalyst structures. [selector.py](/Users/dhyana/dharma_swarm/dharma_swarm/selector.py#L28) [selector.py](/Users/dhyana/dharma_swarm/dharma_swarm/selector.py#L215)
- The strange loop is not fully closed. `reflect_on_cycle()` writes traces, but reflection is not archived as a first-class artifact that conditions the next proposal seed directly. [evolution.py](/Users/dhyana/dharma_swarm/dharma_swarm/evolution.py#L2038)

## Risks
- The highest risk is breaking the very large existing Darwin test surface by renaming public concepts too early. Preserve `DarwinEngine`, `Proposal`, and `FitnessScore` as compatibility adapters until the generic path is proven.
- Archive migration is delicate because ranking, JSONL persistence, and Merkle verification are already tested. Make new fields additive and keep the old weighted-fitness projection stable. [archive.py](/Users/dhyana/dharma_swarm/dharma_swarm/archive.py#L307) [archive.py](/Users/dhyana/dharma_swarm/dharma_swarm/archive.py#L476)
- Meta-instability is real if you simultaneously evolve score weights, selection policy, and vitals response. Gate this behind bounded deltas exactly as the current meta loop already does.
