# DGC Dual-Engine Reality Map

Date: 2026-03-13
Purpose: state plainly where DGC stands, what is real, what is missing, what is frontier-valid, and what to build next if the goal is both semantic depth and world-class execution

## Plain Thesis

DGC is strongest when seen as a `dual-engine mission organism`:

1. a `semantic engine` that metabolizes code, notes, files, and research into structured understanding
2. an `execution engine` that turns hardened understanding into plans, code, artifacts, and live campaigns

The moat is not either engine alone. The moat is the coupling.

If DGC only understands, it becomes an ornate note system.
If DGC only executes, it becomes a strong but forgetful worker swarm.
If it composes both under one campaign ledger, it becomes a genuine new category.

## Four-Column Map

| Surface | What Exists | What Is Missing | What Is Frontier-Valid | What To Build Next |
|---|---|---|---|---|
| `Mission control` | DGC already has real orchestration surfaces in `dgc_cli.py`, `thinkodynamic_director.py`, `task_board.py`, `orchestrator.py`, `session_ledger.py`, and `runtime_state.py`. | A single sovereign mission object is still too weak. Convergence pressure is inconsistent and the director is not yet the commanding default in every serious run. | Persistent multi-agent mission continuity is a real frontier advantage when it survives context loss and keeps shipping artifacts across days. | Make `ThinkodynamicDirector` the default command layer and bind all important tasks to one canonical campaign record. |
| `Semantic digestion` | The repo can already digest `.py`, `.md`, `.txt`, frontmatter, and wikilinks through `semantic_digester.py`, then bridge results into memory through `semantic_memory_bridge.py` and `memory_lattice.py`. | The engine is still narrow relative to a full personal corpus: weak support for PDFs, docs, mail, bookmarks, transcripts, images, OCR, and richer note semantics. | Deterministic local graph construction is exactly the right foundation. This should stay boring, auditable, and cheap. | Build a `corpus adapter layer` with explicit handlers for vaults, repos, documents, exported chats, and research artifacts. |
| `Research grounding` | `semantic_researcher.py` and `field_knowledge_base.py` provide a real external reference spine. | The current research layer is too curated and too static for the broader vault problem. It does not yet behave like a contradiction-hunting research organ. | Hybrid grounding is the right model: local canon first, targeted external research second, provenance everywhere. | Add a `research router` that classifies claims into local-only, local-plus-registry, or live-research-needed buckets. |
| `Semantic synthesis` | `semantic_synthesizer.py`, `semantic_gravity.py`, and the recent PSMV work already show that DGC can identify clusters and cross-corpus intersections. | The output is still mostly scaffolding. Clusters are not yet consistently converted into decision-grade specs or editorially strong synthesis. | Finding the high-gravity intersections in a person's corpus is the right upstream planning function for agentic systems. | Introduce a `cluster-to-spec compiler` that emits build briefs, research memos, contradiction maps, and ranked next actions. |
| `Execution machinery` | DGC has real execution parts: `swarm.py`, `agent_runner.py`, `startup_crew.py`, `plan_compiler.py`, `diff_applier.py`, `evolution.py`, and `full_power_probe.py`. | The execution layer still drifts into generic task churn, duplicate delegation, and autonomy theater under long unattended runs. | A high-discipline worker swarm is valuable if it is evidence-backed, write-safe, and artifact-producing. | Enforce a strict split between read-heavy parallel agents and single-authority write agents, with mandatory verify-and-record loops. |
| `Quality and ethics` | `telos_gates.py`, `anekanta_gate.py`, `diff_applier.py`, and `evolution.py` are unusually strong compared with typical autonomous coding stacks. | These gates are not yet consistently threaded through every semantic-to-build transition. | Ethical and epistemic gates are part of the moat if they are computationally enforced rather than merely described. | Add fail-closed gates at each conversion boundary: digest to claim, claim to cluster, cluster to spec, spec to code, code to archive. |
| `External corpus product thesis` | The PSMV branch proved that DGC can stage a corpus-to-hyperfile pass and construct a cross-corpus bridge. | The hyperfiles are still scaffolds and the output quality is below "industry-grade" or "editorial-grade" synthesis. | The emerging product category is real: compile a person's lifetime corpus into buildable programs and living execution queues. | Shift the external product from "hyperfile generator" to `teleological compiler for personal and institutional knowledge`. |
| `Standing in the AI world` | DGC already combines ingredients most systems keep separate: memory, orchestration, semantics, evaluation, and telos. | It does not yet beat specialist tools on polish, reliability, or final-output quality. | The frontier gap is the bridge from corpus understanding to directed execution. That is where DGC can matter. | Build the bridge so well that repo self-understanding and vault understanding feed one live production loop. |

## What Is Real

The real part is the substrate:

- DGC can parse and index meaningful structure.
- DGC can run real mission-control and swarm machinery.
- DGC can generate artifacts rather than only chat.
- DGC has stronger epistemic and ethical gating than most agent stacks.

That is enough to say this is not fake.

## What Is Not Real Yet

These are still not fully real:

- unattended mission continuity you can trust for many hours
- editorially excellent semantic synthesis at scale
- consistently accurate external research expansion across arbitrary domains
- automatic conversion from a cluster of ideas into a product that ships cleanly

So the right posture is: `promising organism, incomplete command layer`.

## Edge

The edge is `teleological coupling`.

Most systems today do one of these well:

- answer from a corpus
- write or edit code
- search the web
- coordinate a few workers

Very few systems hold all of the following at once:

- provenance
- memory
- architectural self-model
- cross-corpus semantic gravity
- mission continuity
- safe execution

That combination is DGC's real edge if it is forced into artifact production.

## Real Use Cases

The strongest real use cases are:

1. `Founder engine`
   A founder drops years of notes, decks, and repos into the system and gets back ranked venture theses, technical specs, and launch sequences.
2. `Research lab control plane`
   A lab turns papers, notebooks, repos, and experiment logs into a living graph that drives the next experiment and the next software artifact.
3. `Institutional memory compiler`
   A nonprofit, studio, or mission-driven org turns documents plus code into coherent campaigns, briefs, and delivery plans.
4. `Codebase self-understanding`
   Agents stop editing blindly because they can see concept structure, dependencies, and research context before they mutate a system.
5. `Personal teleology engine`
   A serious knowledge worker uses DGC not as a search box but as a machine that says what their corpus is converging toward and what should be built next.

## Design Rule

Keep the substrate deterministic.
Make judgment the expensive layer.

That means:

- parsing should be local, strict, and boring
- provenance should be explicit
- synthesis should be pressure-tested
- execution should be led by evidence, not vibes

## Bottom Line

DGC should not choose between `semantic deepening` and `swarm execution`.

It should become the system where:

- semantic deepening improves what gets built
- execution improves what the system knows
- both are bound by one mission ledger
- both survive context death
- both compound over time

That is the actual category.
