# DGC Self-Proving Studio Swarm

Date: 2026-03-13
Purpose: run a 5-10 agent mission that turns DGC into its own working venture studio and produces the first buyer-ready commercial packet
Primary reference: `docs/reports/DGC_SELF_PROVING_VENTURE_STUDIO_2026-03-13.md`

## Mission Thesis

Do not ask the swarm to "find business ideas."

Ask it to produce the exact packet needed to prove that DGC can run a revenue mission on itself and convert that proof into paid design-partner conversations.

The mission is complete only when a human operator could use the packet immediately to:

- run a demo
- score targets
- send outreach
- close a diagnostic
- deliver the first proof packet

## North Star

Produce one `Working System Packet` for the first offer:

`Campaign X-Ray for AI-native founders`

## Required Outputs

The swarm should produce all of the following:

1. `working_system_spec.md`
   The exact internal operating loop DGC must be able to run on itself.
2. `campaign_xray_spec.md`
   Public promise, scope, exclusions, deliverables, and success criteria.
3. `demo_script.md`
   The shortest persuasive live walk-through.
4. `proof_packet.md`
   Repo artifacts, runtime evidence, and claim boundaries.
5. `target_account_rubric.md`
   Scoring formula and account-selection logic.
6. `target_account_list.md`
   First `25` scored targets from the actual warm graph only.
7. `pricing_sheet.md`
   Design-partner pricing plus standard pricing.
8. `outreach_pack.md`
   Tailored messages for the top target segments.
9. `operator_scoreboard.md`
   The 30-day metrics and stop conditions.
10. `risk_register.md`
   Commercial, delivery, and trust risks.
11. `director_summary.md`
   One-page synthesis with the top next actions.

## Hard Rules

- no invented customer names
- no claims without artifact proof or direct operator evidence
- no promise that depends on fake full autonomy
- no pricing without delivery boundaries
- no demo that hides the brittle parts
- no packet that requires another week of interpretation before use

## Swarm Shape

Minimum viable swarm: `5 agents`

Full studio swarm: `10 agents`

## Roles

1. `Director`
   Owns sequence, coherence, and final packet quality.
2. `Internal Operator`
   Maps the exact DGC internal loop and current runtime truth.
3. `Productizer`
   Sharpens the first offer into a clean product boundary.
4. `Proof Miner`
   Extracts repo and runtime evidence that support the story.
5. `Demo Builder`
   Designs the shortest convincing live narrative.
6. `Buyer Mapper`
   Ranks buyer types and identifies the first target set.
7. `Pricing Strategist`
   Sets design-partner and standard price logic.
8. `Offer Architect`
   Turns the product into a legible commercial page and proposal shape.
9. `Validator`
   Red-teams proof, promise, brittleness, and overclaiming.
10. `Closer`
   Converts the packet into outreach and next-call assets.

## Core Questions

Every agent should answer only the questions relevant to its role.

Shared core questions:

- What exactly is the first product?
- What proof already exists?
- What still breaks?
- Which buyers hurt enough to pay now?
- What is the shortest path to one paid diagnostic?

## Definitions Of Good Output

### Good working-system spec

- grounded in the current repo and runtime
- explicit about where humans are still required
- has clear input, output, and stop conditions

### Good offer design

- one sharp promise
- one sharp boundary
- one defined output packet
- one obvious continuation path

### Good proof packet

- cites real artifacts
- distinguishes prototype proof from production proof
- does not overstate execution reliability

### Good target list

- all targets come from the real warm graph
- each target has a score and rationale
- each target is matched to one offer angle

### Good demo

- starts with a messy corpus
- shows compression into one campaign
- shows one artifact or packet
- ends with the buyer's next decision becoming obvious

## Scoring Formula

Use this account score:

`PartnerScore = 0.30 Pain + 0.20 Urgency + 0.20 CorpusDensity + 0.15 BuyingAuthority + 0.10 ProofLeverage + 0.05 TrustFit`

Use this offer score:

`OfferPriority = WinProbability x ProofValue x ACV / DeliveryRisk`

If an output ignores these formulas, it is too vague.

## Mission Loop

1. map the internal truth
2. package the first product
3. extract proof and claims
4. choose the first targets
5. build the demo and outreach assets
6. red-team the packet
7. prepare for immediate operator use

## Stop Conditions

The swarm must stop when:

- the first product is unambiguous
- the demo can be run live in under `15` minutes
- the first `25` targets are scored
- the packet is sufficient for outreach this week
- the next commercial step is obvious

The swarm must not keep polishing once the packet is commercially usable.

## Failure Modes

This mission fails if:

- agents generate abstract startup language without proof
- target accounts are invented or guessed
- the demo depends on magic autonomy
- pricing is disconnected from delivery labor
- no part of the packet could be used by a human this week

## First Human Actions After Packet Completion

1. choose the top `10` targets
2. tailor the first outreach messages
3. run the live demo once internally
4. send outreach
5. book the first discovery calls
6. refine the packet based on objections
7. close the first paid `Campaign X-Ray`

## Bottom Line

This swarm is not a brainstorm.

It is a bounded commercial mission whose job is to prove that DGC can run a real revenue loop on itself and turn that truth into one more paying customer.
